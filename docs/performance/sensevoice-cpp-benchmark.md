# SenseVoice C++ Benchmark

Date: 2026-06-22

## Summary

This benchmark compares the current EchoFlow Qwen3-ASR C++ path with the
SenseVoiceSmall GGUF C++ runtime from FunAudioLLM/SenseVoice.

Result:

- SenseVoiceSmall C++ is much faster on the tested short dictation samples.
  Across 8 EchoFlow recordings, Qwen averaged 4.21 s after preload, while
  SenseVoice averaged 0.39 s compute time and 0.59 s full CLI wall time.
- On the small labeled sanity set, both engines produced 0 CER on the first two
  samples after punctuation, case, and simplified/traditional Chinese
  normalization. Both produced the same one-character-style error on the third
  sample by recognizing `wav` as `wave`.
- On EchoFlow real recordings, SenseVoice made several clear substitutions that
  Qwen did not, for example `这次` -> `电池`, `上屏` -> `商品`, and
  `不太靠谱` -> `不太够`.
- SenseVoice punctuation/断句 is acceptable for simple short speech, but it
  often under-splits longer EchoFlow recordings. Qwen generally placed question
  marks and clause boundaries more naturally on these samples.

Recommendation: SenseVoice is worth integrating as an optional benchmark or
low-latency backend, but this dataset does not justify replacing Qwen as the
default final-text engine yet.

## Runtime Setup

Qwen path:

```bash
cmake --build build --target echoflow-service voice_latency_benchmark
./build/service/echoflow-service --self-test
```

SenseVoice path:

```bash
git clone --depth 1 https://github.com/FunAudioLLM/SenseVoice.git \
  /tmp/echoflow-sensevoice-eval/SenseVoice

curl -L -o /tmp/echoflow-sensevoice-eval/llama.cpp-8086439.tar.gz \
  https://github.com/ggml-org/llama.cpp/archive/8086439a4cea94c71a5dfb8fe4ad1546aebd640f.tar.gz

tar -xzf /tmp/echoflow-sensevoice-eval/llama.cpp-8086439.tar.gz \
  -C /tmp/echoflow-sensevoice-eval/llama-src --strip-components=1

cmake -S runtime/llama.cpp \
  -B /tmp/echoflow-sensevoice-eval/sensevoice-build-local \
  -DCMAKE_BUILD_TYPE=Release \
  -DGGML_NATIVE=ON \
  -DLLAMA_CURL=OFF \
  -DFETCHCONTENT_SOURCE_DIR_LLAMA=/tmp/echoflow-sensevoice-eval/llama-src

cmake --build /tmp/echoflow-sensevoice-eval/sensevoice-build-local -j$(nproc)
```

Models:

```bash
env -u all_proxy -u ALL_PROXY uvx --from huggingface_hub hf download \
  FunAudioLLM/SenseVoiceSmall-GGUF \
  --include 'sensevoice-small-q8.gguf' \
  --local-dir /tmp/echoflow-sensevoice-eval/models

env -u all_proxy -u ALL_PROXY uvx --from huggingface_hub hf download \
  FunAudioLLM/fsmn-vad-GGUF \
  --include '*.gguf' \
  --local-dir /tmp/echoflow-sensevoice-eval/models
```

Files used:

- `sensevoice-small-q8.gguf`: 254,208,320 bytes
- `fsmn-vad.gguf`: 1,720,512 bytes

The upstream prebuilt Linux x64 release binary was also tried, but it exited
with status 132 on this host. The local build was required to avoid an illegal
instruction fault.

## Methodology

Latency:

- Qwen latency uses `build/tests/voice_latency_benchmark` with model preload and
  3 iterations per file. The table reports the average of iterations 2 and 3 to
  reduce first-run noise after preload.
- SenseVoice latency uses the C++ GGUF CLI with `--vad fsmn-vad.gguf`.
  `compute_ms` comes from SenseVoice's own `[sensevoice] done` timer, which
  starts after model loading. `wall_ms` measures the full CLI process, including
  model loading.
- SenseVoice uses 8 CPU threads internally in the current upstream C++ source.
  Qwen used the current EchoFlow default `openblas_threads=4`.

Accuracy:

- The 8 EchoFlow recordings do not have human reference transcripts, so they are
  not used for numeric CER.
- Numeric CER is only reported for labeled samples with known reference text.
- CER normalization removes punctuation and whitespace, uppercases Latin text,
  and normalizes the few traditional Chinese characters present in the labeled
  sample references to simplified Chinese.

## Accuracy And Text Quality Notes

The real EchoFlow recordings do not have reference transcripts, so the following
are qualitative observations from side-by-side output rather than CER:

- SenseVoice is clearly faster, but it made several user-visible substitutions:
  `这次` -> `电池`, `上屏` -> `商品`, `不太靠谱` -> `不太够`, and `采分/分段`
  confusion.
- Qwen also made likely errors, for example `先是是默认的音响`, but it generally
  preserved the intended clause structure and question boundaries better.
- SenseVoice's VAD returned 0 segments for the `嗯` sample, so a production
  integration needs a fallback for short interjections or VAD-empty output.

## 断句 Observations

Qwen:

- More often emits Chinese question marks at natural question boundaries.
- Often strips the final punctuation in EchoFlow because the current user config
  has `basic.recognition.strip_trailing_punctuation=true`.
- Better on multi-clause recordings where a question is followed by a statement.

SenseVoice:

- Emits sentence-final punctuation in most non-empty cases.
- Often keeps a longer phrase as one comma-linked sentence where Qwen separates
  a question.
- Relies on FSMN-VAD segmentation; in these samples, VAD usually returned 1
  segment even when the text naturally contained more than one sentence.
- Needs explicit handling for VAD returning no speech.

## Decision

SenseVoice C++/GGUF is a strong candidate for an optional low-latency backend.
The speed advantage is large enough to justify a real integration experiment.

It should not replace Qwen as the default final-text engine from this benchmark
alone. The measurable labeled set is too small, and the EchoFlow real recordings
show enough SenseVoice substitution and under-segmentation risk that the safer
product path is:

1. Add SenseVoice behind an engine switch or benchmark-only adapter.
2. Keep Qwen as the default final commit engine.
3. Expand the labeled Chinese test set with real EchoFlow-style phrases.
4. Re-run CER plus human segmentation review before changing the default.
